   `"add"` in `tests/` surfaced 5 legitimate broad-staging fixture calls

   state flip (CI green → CodeRabbit landed → fix round green) with no

### What Was Surprising

1. **The task spec itself carried the imprecision the convention exists to
   prevent** — KIT-0037's goal text (copied from the retro) lumped "can't
   fetch" into exit 2, but the reference implementation routes source
   acquisition to exit 4. Codifying a convention means checking the
   convention's own wording against the code it cites, not just the code
   against the convention.
2. **Preflight Gate 6 does an exact-name match** (`<TASK>-REVIEW-STARTER.md`,
   `preflight-check.sh:487`) — a bundle's combined
   `KIT-0037-38-39-REVIEW-STARTER.md` failed the gate. Needed a rename to the
   lead task plus per-task pointer files (starters and evaluator records) so
   all three tasks pass preflight independently.
3. **CodeRabbit reviewed docs-only content substantively** — it ran its own
   verification scripts against `resolve_source()` to check the claimed exit
   codes. Bot review of prose that makes factual claims about code is worth
   having even when the planner skips the adversarial evaluator.

### What Should Change

1. **Preflight should support task bundles** — either Gate 5/6 accept a
   shared artifact naming pattern (e.g. glob `<TASK>*REVIEW-STARTER.md`) or
   the review-handoff docs codify the lead-task + pointer-files pattern used
   here. Today the requirement is discoverable only by failing the gate.
2. **Specs that cite exit codes should quote the code, not the retro** — same
   family as the existing "spec-verified API shapes" planner rule: when a
   task spec asserts runtime behavior (exit codes, event names, API shapes),
   paste the verified source line, not a prose summary from a retro.
